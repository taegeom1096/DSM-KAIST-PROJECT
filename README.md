# 메뉴 추천 웹사이트 README

## 1. 팀명과 서비스 이름

- 팀명: 메뉴 추천 팀
- 팀원: 김태건, 조현우
- 서비스 이름: 오늘 뭐 먹지? - 메뉴 추천 웹사이트
- 한 줄 소개: 배고픈 사용자가 메뉴를 오래 고민하지 않도록 버튼 한 번으로 오늘의 메뉴를 추천해주는 서비스입니다.

## 2. 사용자는 누구인가

이 서비스의 주요 사용자는 점심이나 저녁 메뉴를 쉽게 결정하지 못하는 사람입니다.

- 메뉴 선택에 시간이 오래 걸리는 사람
- 매번 비슷한 음식을 먹어서 새로운 추천이 필요한 사람
- 배달 앱이나 식당 목록을 보며 오래 고민하는 사람
- 간단하고 재미있게 메뉴를 정하고 싶은 학생 또는 자취생

## 3. 왜 필요한 서비스인가

메뉴를 고르는 일은 단순해 보이지만, 실제로는 생각보다 시간이 오래 걸립니다. 배가 고픈 상태에서 여러 선택지를 비교하다 보면 결정이 더 어려워질 수 있습니다.

이 서비스는 사용자가 직접 많은 메뉴를 비교하지 않아도 서버가 등록된 메뉴 중 하나를 랜덤으로 추천해줍니다. 사용자는 추천 결과를 저장하고, 마음에 들지 않으면 다시 뽑을 수 있습니다. 또한 메뉴 후보를 직접 추가하고 한 줄 평을 남길 수 있어, 사용할수록 추천 후보와 기록이 쌓이는 장점이 있습니다.

## 4. 핵심 기능 데모

### 1. 메뉴 추천받기

1. 사용자 이름을 입력합니다.
2. `음식 뽑기` 버튼을 클릭합니다.
3. 서버가 메뉴 목록 중 하나를 랜덤으로 선택합니다.
4. 화면에 추천 메뉴 이름, 카테고리, 칼로리, 맵기, 평균 별점이 표시됩니다.

### 2. 추천 기록 관리

- 최근 추천 기록을 목록으로 확인할 수 있습니다.
- 추천 기록에서 `상세`, `다시`, `삭제` 기능을 사용할 수 있습니다.
- `다시` 버튼을 누르면 기존 추천 기록의 메뉴가 새 메뉴로 변경됩니다.

### 3. 메뉴 후보 등록

- 메뉴 이름, 카테고리, 칼로리, 맵기, 등록자를 입력해 새 메뉴를 추가할 수 있습니다.
- 추가한 메뉴는 바로 랜덤 추천 후보에 포함됩니다.

### 4. 한 줄 평 작성

- 추천 결과 상세 화면에서 메뉴에 대한 한 줄 평과 별점을 남길 수 있습니다.
- 등록된 한 줄 평은 메뉴의 평균 별점과 리뷰 개수에 반영됩니다.

## 5. 서버/API/DB/프론트 구조 설명

### 서버 구조

서버는 Node.js 기본 `http` 모듈로 구현되어 있습니다. `server.js`가 정적 파일 제공과 API 요청 처리를 모두 담당합니다.

- `GET` 요청으로 HTML, CSS, JavaScript 파일 제공
- `/menus`, `/recommendations`, `/reviews` API 처리
- 요청 본문 JSON 파싱
- 에러 응답 처리
- `data/db.json` 파일 읽기/쓰기

### API 구조

| Method | Path | 설명 |
| --- | --- | --- |
| GET | `/health` | 서버 상태 확인 |
| GET | `/menus` | 메뉴 목록 조회 |
| POST | `/menus` | 추천 후보 메뉴 등록 |
| GET | `/recommendations` | 추천 기록 목록 조회 |
| POST | `/recommendations` | 랜덤 메뉴 추천 생성 |
| GET | `/recommendations/:id` | 추천 기록 단건 조회 |
| PATCH | `/recommendations/:id` | 추천 메뉴 다시 뽑기 |
| DELETE | `/recommendations/:id` | 추천 기록 삭제 |
| GET | `/menus/:id/reviews` | 메뉴 한 줄 평 목록 조회 |
| POST | `/menus/:id/reviews` | 메뉴 한 줄 평 작성 |
| DELETE | `/reviews/:id` | 한 줄 평 삭제 |

### DB 구조

별도 데이터베이스 서버 대신 `data/db.json` 파일에 데이터를 저장합니다.

- `menus`: 추천 후보 메뉴 정보
- `recommendations`: 사용자가 추천받은 기록
- `reviews`: 메뉴별 한 줄 평과 별점
- `counters`: 새 데이터 생성을 위한 ID 관리

메뉴 데이터에는 이름, 카테고리, 칼로리, 맵기, 등록자, 생성일이 저장됩니다. 추천 기록은 사용자 이름, 추천 방식, 추천된 메뉴 ID, 생성일과 수정일을 저장합니다.

### 프론트 구조

프론트는 `public` 폴더에 있는 정적 파일로 구성되어 있습니다.

```text
public/
├── index.html
├── styles.css
└── app.js
```

- `index.html`: 화면 구조와 입력 폼, 탭 영역 구성
- `styles.css`: 전체 레이아웃과 카드, 버튼, 탭 스타일 정의
- `app.js`: API 호출, 화면 렌더링, 버튼 이벤트 처리

### 전체 폴더 구조

## 6. 가장 어려웠던 문제와 해결 과정

가장 어려웠던 부분은 프론트 화면의 여러 기능과 서버 데이터를 자연스럽게 연결하는 것이었습니다. 메뉴 추천, 추천 기록, 상세 화면, 한 줄 평 기능이 서로 연결되어 있기 때문에 한 기능을 실행한 뒤 다른 화면의 데이터도 함께 갱신해야 했습니다.

예를 들어 메뉴를 다시 추천하면 추천 결과 카드뿐만 아니라 추천 기록 목록과 상세 화면도 같이 바뀌어야 합니다. 이를 해결하기 위해 프론트의 `state` 객체에 현재 추천 목록, 메뉴 목록, 선택된 추천 기록, 한 줄 평 목록을 저장하고, 기능 실행 후 필요한 데이터를 다시 불러오도록 구성했습니다.

또 다른 어려움은 JSON 파일을 데이터 저장소처럼 사용하는 것이었습니다. 서버에서 `data/db.json`을 읽고 수정한 뒤 다시 저장해야 했기 때문에, 데이터 구조를 `menus`, `recommendations`, `reviews`, `counters`로 나누어 관리했습니다. 새 데이터를 만들 때는 `counters` 값을 사용해 ID가 겹치지 않도록 처리했습니다.

- 카테고리별 메뉴 필터링 기능 추가
- 칼로리와 맵기 기준으로 추천 조건 설정
- 메뉴 이미지 등록 기능 추가
- 추천 결과 공유 기능 추가

- 한 줄 평 수정 기능 추가
- 모바일 화면에서 더 보기 좋은 UI 개선

## 좋았던 점

- Node.js만으로 서버, API, 정적 파일 제공을 모두 구현하면서 백엔드 흐름을 이해할 수 있었습니다.
- 프론트에서 `fetch`를 사용해 API와 직접 통신하며 풀스택 구조를 경험할 수 있었습니다.
- 추천 기록, 메뉴 관리, 한 줄 평 기능이 연결되어 실제 서비스처럼 동작하는 점이 좋았습니다.

## 사용 중 헷갈린 점

- API 요청 후 어떤 화면 데이터를 다시 불러와야 하는지 정리하는 부분이 헷갈렸습니다.
- 추천 기록과 메뉴, 한 줄 평 데이터가 서로 연결되어 있어 ID 관계를 맞추는 부분이 어려웠습니다.
- JSON 파일을 수정한 뒤 저장하는 방식이 실제 DB와 달라 처음에는 데이터 흐름을 이해하는 데 시간이 걸렸습니다.

## 기능/UX 개선 제안

- 추천 버튼을 눌렀을 때 슬롯머신처럼 메뉴가 빠르게 바뀌는 애니메이션을 추가하면 더 재미있을 것 같습니다.
- 사용자가 원하는 카테고리만 선택해서 추천받을 수 있으면 더 실용적일 것 같습니다.
- 메뉴별 이미지를 보여주면 추천 결과가 더 직관적으로 보일 것 같습니다.
- 삭제 버튼을 누르기 전에 확인 창을 추가하면 실수로 기록을 삭제하는 일을 줄일 수 있습니다.

## 버그 또는 재현 절차

- 현재 확인된 치명적인 버그는 없습니다.
- 메뉴 후보가 0개일 때 추천을 요청하면 서버가 `추천할 메뉴가 없습니다. 먼저 메뉴를 등록해주세요.`라는 에러를 반환합니다.
- 특정 메뉴의 평을 보려면 해당 메뉴를 추천받은 기록이 필요합니다. 메뉴 목록에서 바로 평을 볼 수 있도록 개선할 수 있습니다.
